REST (Representational State Transfer)

Core REST Principles

  • Client-Server Architecture

    Client (frontend) and server (backend) are separate

    Client handles UI; server handles data and logic

    👉 Makes systems easier to scale and maintain

  • Statelessness

    Each request contains all the information needed

    Server does not store client session state

    Example: Each request have Authorization token and all

  • Cacheable

    Responses should indicate if they can be cached

    Improves performance and reduces server load

  • Uniform Interface

    Everything is a resource (user, product, order)

    Identified via URL, Ex: /users/123

    Resource Representation in JSON

    Requests/responses contain enough info

  • Layered System

    Client doesn't know if it's connected to server, proxy or load balancer?

  • Code on Demand (optional)

    Server can send executable code (rare in practice)

What is a REST API?

A REST API is an API that follows REST principles and uses HTTP methods.

It allows communication between

Frontend ↔ Backend

Apps ↔ Servers

Services ↔ Services

Common HTTP Methods

GET ↔ Retrieve data

POST ↔ Create resource

PUT ↔ Update entire resource

PATCH ↔ Partial update

DELETE ↔ Remove resource

HTTP Status Codes

200 ↔ OK

200 ↔ OK

201 ↔ Created

400 ↔ Bad Request

401 ↔ Unauthrized

404 ↔ Not Found

500 ↔ Server Error

Authentication in REST APIs

API Keys

OAUTH 2.0

JWT (JSON Web Tokens)

Key Concepts in REST APIs

Resource: Any object (user, product, etc.)

Endpoint: URL where resource lives:

Representation: JSON or XML

Headers: Metadata (Auth, Content type)
